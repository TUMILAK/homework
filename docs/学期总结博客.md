## 一、项目介绍

### 搜题 Agent — 一个基于 AI 的搜题/学习辅助 Web 应用

**核心功能：**

| 功能 | 说明 |
|------|------|
| 🤖 多模型支持 | DeepSeek / OpenAI / Anthropic / 自定义，前端切换 |
| 📷 图片搜题 | 上传图片 → PaddleOCR 本地识别 → AI 分步解答 |
| 💬 聊天模式 | 搜题 / 自由聊天双模式 |
| 📁 本地文件 | 支持 TXT/MD/PDF，递归读取文件夹 |
| 📝 回答存档 | 按日期自动存 `answers/YYYY-MM-DD.md` |

**技术栈：**
- 后端：FastAPI + Uvicorn
- AI：OpenAI SDK（兼容多模型）
- OCR：PaddleOCR（纯本地）
- 前端：原生 HTML/CSS/JS + Pico CSS 深色主题

## 二、项目部署

### GitHub 仓库
🔗 https://github.com/TUMILAK/homework

### 本地运行

```bash
# 1. 克隆项目
git clone https://github.com/TUMILAK/homework.git
cd homework

# 2. 安装依赖
pip install -r requirements.txt

# 3. 启动服务
python run_server.py
```

浏览器打开：**http://127.0.0.1:8010**

### 环境变量（可选）

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `SOUTI_PORT` | `8010` | 监听端口 |

| `PADDLE_OCR_USE_GPU` | `false` | GPU 加速 |

> ⚠️ PaddleOCR 首次运行需联网下载模型。Windows CPU 用户请参考 README 中的 Paddle 安装说明。

## 三、学期回顾

### 3.1 回顾你对于软件工程课程的想象

开学时，我对这门课的期待很简单：**学会写一个完整的项目**。

在此之前，代码对我来说就是课程作业里的几个 `.py` 文件——写完交上去，从此再不见。我不懂什么是"项目结构"，不知道 Git 除了 `git push` 还能干什么，更没想过要把一个想法从零变成用户能真正打开用的东西。

**达到期待的方面：**
- ✅ 独立完成了一个完整的 Web 应用（后端 API + 前端界面 + OCR + AI 调用）
- ✅ 学会了 Git 版本控制（虽然一开始的 commit message 很不堪入目）
- ✅ 理解了前后端分离的基本架构
- ✅ 掌握了 FastAPI、PaddleOCR 等技术的实际运用
- ✅ 体验了发布 Release、写文档的完整流程

**仍然不足的方面：**
- ❌ 前端仍停留在原生 HTML/CSS/JS，没有学 React/Vue（一方面是项目规模不需要，另一方面也确实没时间）
- ❌ 测试几乎为零——整个项目没有写过一行自动化测试
- ❌ 部署仅限本地运行，没有上云（买不起服务器）
- ❌ CI/CD、Docker 容器化等 DevOps 实践完全空白

**为什么？** 两个人要在三个月里做完后端、前端、OCR 集成、AI 接口、文档和答辩，能把功能跑通已经拼尽全力。"先让它能用，再让它好用"——这是我们给自己找的借口，也算是一句实话。

### 3.3 令你印象最深刻的是哪一次作业或哪一场答辩？

**最深刻：OCR 识别 Bug 修复（2026.06.14）**

就在项目已经"完成"、Release Notes 都写好的时候，一个致命的 Bug 出现了：**PaddleOCR 在 Windows 上只要识别图片就崩溃**。

错误信息长这样：

```
fused_conv2d error at ...\paddle\phi\kernels\onednn\...
```

百度、Google、GitHub Issues 翻了个遍，答案五花八门：有人说重装 Paddle、有人说换 Linux、有人说退回到 Paddle 2.x……但我们不能退回去，因为新版本的依赖已经锁死了 Paddle 3.3。

最后我们发现：**Paddle 3.3 在 Windows CPU 上即使显式设置 `enable_mkldnn=False`，IR 图融合阶段仍然会走 OneDNN 路径**。唯一的解法是 monkey-patch Paddle 的底层 `Config` 类。

我们在 `ocr_service.py` 里写了三层防护：

1. **环境变量预设置**：在所有 import 之前关掉 `FLAGS_use_onednn` 等
2. **monkey-patch Inference Config**：强制 `switch_ir_optim(False)`、禁用 MKLDNN
3. **Paddle runtime flags**：在引擎初始化时用 `paddle.set_flags()` 二次确认

最后 commit 的那条消息是：`fix: Paddle 3.3 OCR crash — patch OneDNN/IR fusion for Windows CPU`

那一刻，我真正理解了什么叫"**知其然，还要知其所以然**"。

## 四、总结收获

### 4.1 展开说说你的软工实践故事

**第一阶段：从零到"111"（3月中 ~ 3月底）**

故事的开头很朴素——`Initial commit`，连个像样的 README 都没有。接下来的几天里，我们上传了自我介绍、加了个 Logo、把项目结构搭了出来。但说实话，那时候根本不知道自己在做什么。那条 `111` 的 commit message 至今让我脸红：它就是随手打的，因为当时连给代码起个描述都嫌麻烦。

转折出现在 3 月 30 日。那天我们终于写出了一份 `requirements.txt`，第一次认真思考"这个项目到底依赖什么"。同一天我们还经历了一次激烈的讨论——后端觉得前端进度太慢，前端觉得后端需求太多。吵完之后，我们重新划分了任务边界，并约定了接口先行、文档同步的工作方式。那次争吵没有让项目崩掉，反而让后续的合作顺畅了许多：**分歧不可怕，分歧之后还能坐下来继续写代码才重要**。

**第二阶段：摸索与成型（4月 ~ 5月）**

4 月 16 日的 `new` 和 5 月 10 日的 `0.1`，标志着项目从"一团乱麻"变成了"有个样子"。我们实现了基本的前后端通信、模型调用、文件读取。5 月 24 日的 `ppt` 是答辩前的紧急冲刺，5 月 25 日居然还顺手加了个天气查询功能（`feat:weather api`）——典型的"正经事不做，净整花活"。

**第三阶段：收官与补刀（6月）**

6 月 9 日是项目的"大日子"——我们把所有的零散文件做了一次大整理，写了一份 300 行的详细 README，发布了 v1.0.0。然后，正如前文所说，6 月 14 日被 Paddle 3.3 的 Bug 捅了一刀，又爬起来补了一刀。

**一点感悟：**

回顾这 15 次提交，代码本身不一定多优秀，但这条时间线记录了一个完整的"从零到一"。更重要的是，它留下了我们的真实状态——有摸鱼的、有爆炸的、有临时抱佛脚的、也有认真推敲的。这不就是软工实践该有的样子吗？

### 4.2 学习到的新技术或生产力工具

| 技术/工具 | 带来的帮助 |
|-----------|-----------|
| **FastAPI** | 第一次接触现代 Python Web 框架。自动生成 API 文档、类型提示、异步支持——写完就再也不想用 Flask 了。 |
| **PaddleOCR** | 从安装到崩溃到 patch 源码，对 Paddle 的底层推理机制有了深入理解。虽然过程痛苦，但这部分知识在别处学不到。 |
| **OpenAI Python SDK** | 学会了如何用统一的接口调用 DeepSeek、OpenAI、Anthropic 等多个模型，理解了 API 兼容层的重要性。 |
| **Git / GitHub** | 从只会 `push` 到会用 `rebase`、`force push`、`cherry-pick`、`fetch` 处理历史。最值钱的是学会了写有意义的 commit message（虽然后面的比前面的好很多）。 |
| **Pico CSS** | 一个极简的 CSS 框架，不需要学 Tailwind 也能做出像样的深色主题界面。非常适合原型项目。 |
| **Markdown 文档** | README、RELEASE、这篇总结博客——全都是 Markdown。它让你专注于内容，而不是排版。 |

### 4.3 技术之外，这门课程还给你带来了哪些方面的提升？

1. **心态上的成长**：以前遇到报错第一反应是"完了，做不出来了"。现在第一反应是"让我看看日志说什么"——面对未知问题的恐惧感降低了。

2. **团队协作的体感**：两个人做项目，沟通不畅就是灾难。我们经历过互相指责的阶段，但后来学会了明确分工（后端 API 先给接口文档、前端照着文档写）、定期同步（虽然有时候就是 QQ 上吼一句）。

3. **文档意识**：以前觉得代码写完了就行了，现在知道没有 README 的项目等于不存在。一份好的文档不仅给别人看，更是给自己看的——两个月后你自己都忘了这代码是干嘛的。

4. **项目节奏感**：三个月做一个项目，什么时候该冲刺、什么时候该打磨、什么时候该封板——这是课本上学不到的"软技能"。

### 4.4 还想说点什么

如果有什么遗憾的话，那就是**前端真的太丑了**。

我们的搜题 Agent 功能齐全，但界面确实寒酸——原生 HTML/CSS/JS，没有动画、没有响应式、没有加载骨架屏。每次在答辩上看到别的组用 React + Tailwind 做的绚丽界面，都默默低下头。

但转念一想，**这就是真实世界的软件工程**：资源有限、时间不够、技术栈不全，你能做的就是在现有条件下交出最好的东西。我们的 OCR 识别补丁，可能比一个漂亮的按钮更有技术含量。

给未来学弟学妹的一句话：**别怕 commit message 写得丑，怕的是你一条都不写**。那些 `111`、`new`、`0.1` 会提醒你，你确实是从那里出发的。
